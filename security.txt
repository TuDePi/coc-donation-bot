EXECUTIVE SUMMARY
-----------------
The Clash of Clans bot has received several important security fixes (CRIT-001
through CRIT-003) that addressed the most critical prior vulnerabilities: hardcoded
session secrets, ADB command injection via the coordinate API, and path traversal in
config file access. The remaining attack surface still contains significant weaknesses
across authentication session management, CORS policy, CSRF protection, the CoC API
proxy, strategy file path handling, and supply-chain dependency pinning. The most
immediately exploitable remaining issues are the wildcard CORS configuration combined
with missing CSRF tokens (which together allow cross-site request forgery against all
state-changing endpoints) and the unauthenticated strategy-name injection that writes
arbitrary files to the strategies directory.

[INFO-001] USERS.JSON IS TRACKED BY GIT HISTORY
Severity:    INFORMATIONAL
File(s):     users.json, .gitignore
Category:    Credentials / Version Control

Description:
  users.json is listed in .gitignore, which prevents it from being committed in
  future pushes, but the file currently exists on disk and has been committed in
  at least one past revision. Password hashes committed to git history remain
  accessible to anyone who can clone the repository, even after the file is added
  to .gitignore. The current on-disk file contains a real username ("tudepi") and
  a pbkdf2:sha256 hash that could be subject to offline dictionary attacks.

Evidence:
  /Users/egeiltir/Desktop/coc bot/users.json:
    {"username": "tudepi", "password_hash": "pbkdf2:sha256:1000000$..."}
  .gitignore line 11: users.json   (added after initial commit)

Impact:
  Offline dictionary / rainbow-table attack against the stored password hash if
  the repository is ever made public or accessed by an unauthorised party. Low
  probability for a local-only bot, but real risk if the repo is pushed to GitHub.

Remediation:
  1. Run: git filter-repo --path users.json --invert-paths
     (or BFG Repo Cleaner) to purge all history.
  2. Change the account password immediately after cleaning history.
  3. Add a pre-commit hook that refuses to commit files matching users.json.

References:
  CWE-312: Cleartext Storage of Sensitive Information
  https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/removing-sensitive-data-from-a-repository

[INFO-002] WEAK DEFAULT SECRET FALLBACK -- SESSION RESET ON EVERY RESTART
Severity:    INFORMATIONAL
File(s):     web/app.py (lines 32-41)
Category:    Session Management

Description:
  When COC_BOT_SECRET is not set in the environment, a fresh random secret is
  generated at startup. This is safer than a hardcoded default, but means every
  server restart invalidates all active sessions. More importantly, it silently
  degrades security: an operator who does not notice the warning log will run
  the service indefinitely without a stable, persistent secret, making all
  sessions short-lived and potentially causing confusion.

Evidence:
  _secret = os.environ.get("COC_BOT_SECRET")
  if not _secret:
      _secret = secrets.token_hex(32)
      logger.warning("COC_BOT_SECRET is not set...")

Impact:
  No direct exploitability; however, absent a properly set environment variable,
  sessions break on restart, and users may inadvertently re-authenticate over an
  unencrypted connection.

Remediation:
  Document COC_BOT_SECRET as a required environment variable in README/setup
  instructions. Optionally raise a hard error (sys.exit) rather than silently
  generating an ephemeral key, so operators cannot accidentally skip the step.

References:
  CWE-798: Use of Hard-coded Credentials (inverse: failure to enforce externally
  supplied secret)

[INFO-003] UNENCRYPTED ADB COMMUNICATION CHANNEL
Severity:    INFORMATIONAL
File(s):     bot/adb_controller.py
Category:    Transport Security

Description:
  All ADB commands (screenshots, taps, swipes, monkey) are sent over the standard
  ADB TCP/USB channel without additional encryption. If the device is accessed over
  TCP/IP (adb connect <ip>) rather than USB, the entire ADB session -- including
  PNG screenshot data -- is transmitted in plaintext on the local network. ADB does
  authenticate connections via RSA key pairs, but the data channel is not encrypted.

Evidence:
  All _run() calls use subprocess.run(["adb", ...]) with no TLS wrapper.
  The relog function calls monkey: self.adb._run("shell", "monkey", "-p", ...)

Impact:
  On a shared or wireless network, passive traffic capture reveals full device
  screenshots (including any game credentials or payment screens visible in the
  game), exact tap coordinates, and the package name being automated.

Remediation:
  Use USB-only mode when possible. If TCP ADB is required, restrict it to
  localhost (adb -L tcp:localhost:5037) and use an SSH tunnel. Document this
  constraint for users.

References:
  CWE-319: Cleartext Transmission of Sensitive Information

[INFO-004] NO RATE LIMITING ON LOGIN ENDPOINT
Severity:    INFORMATIONAL
File(s):     web/app.py (lines 75-88)
Category:    Authentication

Description:
  The /login POST endpoint performs no request rate limiting, account lockout, or
  CAPTCHA. An attacker with network access can submit unlimited credential attempts,
  bounded only by network speed and server CPU. While this is mitigated by the
  dashboard typically running on localhost/LAN, it remains a concern for any
  deployment exposed to a broader network.

Evidence:
  @app.route("/login", methods=["GET", "POST"])
  def login():
      ...
      if username == user["username"] and check_password_hash(user["password_hash"], password):

  No rate-limit decorator, no failed-attempt counter, no lockout.

Impact:
  Offline and online brute-force of the single admin account. pbkdf2:sha256 with
  1,000,000 iterations provides significant resistance, but unlimited online
  attempts with a weak password would eventually succeed.

Remediation:
  Add flask-limiter (or equivalent) to enforce a limit of e.g. 10 login attempts
  per IP per 10 minutes:
    from flask_limiter import Limiter
    limiter = Limiter(app, key_func=get_remote_address)
    @limiter.limit("10/10 minutes")
    @app.route("/login", methods=["GET", "POST"])

References:
  CWE-307: Improper Restriction of Excessive Authentication Attempts
  OWASP A07:2021 - Identification and Authentication Failures

[LOW-001] SESSION COOKIE MISSING SECURITY ATTRIBUTES
Severity:    LOW
File(s):     web/app.py
Category:    Session Management

Description:
  Flask session cookies are issued without explicitly configuring the Secure,
  HttpOnly, or SameSite cookie attributes. While Flask sets HttpOnly by default,
  the Secure flag (HTTPS-only) and SameSite are not enforced. If the dashboard is
  ever deployed with HTTPS, cookies would still be sent over HTTP, and SameSite
  omission partially contributes to CSRF risk.

Evidence:
  app = Flask(...)
  # No app.config["SESSION_COOKIE_SECURE"] = True
  # No app.config["SESSION_COOKIE_SAMESITE"] = "Lax"
  # No app.config["SESSION_COOKIE_HTTPONLY"] = True  (Flask defaults True, but
  #   explicit is better)

Impact:
  Cookie theft via network sniffing if deployed over HTTP (low probability for a
  local bot), and a reduced barrier for CSRF attacks.

Remediation:
  Add to web/app.py after SECRET_KEY is set:
    app.config["SESSION_COOKIE_SECURE"] = True
    app.config["SESSION_COOKIE_HTTPONLY"] = True
    app.config["SESSION_COOKIE_SAMESITE"] = "Lax"

References:
  CWE-614: Sensitive Cookie in HTTPS Session Without "Secure" Attribute
  OWASP: Session Management Cheat Sheet

[LOW-002] LOGOUT ROUTE ACCEPTS GET REQUESTS -- CSRF LOGOUT
Severity:    LOW
File(s):     web/app.py (lines 110-113), web/templates/index.html (line 17)
Category:    CSRF

Description:
  The /logout route uses only @app.route("/logout") without methods=["POST"],
  meaning it accepts GET requests. The HTML template submits logout as a POST form,
  but the route handler does not enforce POST, so a simple <img src="/logout"> tag
  on any page a logged-in user visits would silently log them out (CSRF logout).
  Note: the broader CSRF issue is also captured in MED-001.

Evidence:
  @app.route("/logout")   # No methods=["POST"] restriction
  def logout():
      session.clear()
      return redirect(url_for("login"))

  index.html line 17:
  <form action="/logout" method="POST">

Impact:
  Any webpage visited while authenticated can force a logout, causing a mild denial
  of service to the operator.

Remediation:
  @app.route("/logout", methods=["POST"])
  @login_required
  def logout():
      session.clear()
      return redirect(url_for("login"))

References:
  CWE-352: Cross-Site Request Forgery
  OWASP A01:2021 - Broken Access Control

[LOW-003] UNPINNED DEPENDENCY VERSIONS ALLOW SILENT UPGRADES TO MALICIOUS PACKAGES
Severity:    LOW
File(s):     requirements.txt
Category:    Dependency / Supply Chain

Description:
  All eight dependencies use >= version constraints with no upper bound or hash
  pinning. This means a pip install --upgrade or a fresh install after a supply-
  chain compromise of any upstream package would silently pull in a malicious
  version. The most risk-relevant packages are flask, flask-socketio (web server),
  opencv-python (binary wheel with C extensions), and pytesseract (wraps an
  external Tesseract binary).

Evidence:
  flask>=3.0.0
  flask-socketio>=5.3.0
  opencv-python>=4.8.0
  numpy>=1.24.0

Impact:
  In a supply-chain attack scenario, a compromised package could execute arbitrary
  code during import or at installation time.

Remediation:
  Pin to exact versions (==) in production or use pip-compile to generate a
  requirements.txt with hashes:
    pip-compile --generate-hashes requirements.in
  At minimum, use compatible-release constraints (~=):
    flask~=3.0.3
    flask-socketio~=5.3.6

References:
  CWE-1395: Dependency on Vulnerable Third-Party Component
  OWASP A06:2021 - Vulnerable and Outdated Components

[LOW-005] ALLOW_UNSAFE_WERKZEUG=TRUE IN PRODUCTION SERVER INVOCATION
Severity:    LOW
File(s):     web/app.py (line 450)
Category:    Configuration

Description:
  socketio.run(..., allow_unsafe_werkzeug=True) suppresses flask-socketio's
  warning about using Werkzeug's development server in production. This flag
  signals that the operator intentionally runs a development-grade HTTP server
  rather than a production WSGI server (gunicorn, uvicorn, etc.). The Werkzeug
  development server is single-threaded by default, not hardened, and lacks
  proper production timeout/connection handling.

Evidence:
  web/app.py line 450:
    socketio.run(app, host=host, port=port, debug=debug, allow_unsafe_werkzeug=True)

Impact:
  Denial-of-service via connection exhaustion is more likely than with a production
  server. No direct code execution risk, but reduced resilience.

Remediation:
  Use eventlet or gevent with gunicorn for production:
    gunicorn --worker-class eventlet -w 1 "web.app:app"
  Remove allow_unsafe_werkzeug=True and the debug parameter from the run() call.

References:
  Flask-SocketIO docs: Deployment

[MED-001] NO CSRF PROTECTION ON ANY STATE-CHANGING API ENDPOINT
Severity:    MEDIUM
File(s):     web/app.py (lines 174-359), web/templates/index.html
Category:    CSRF

Description:
  None of the POST endpoints (/api/start, /api/stop, /api/config, /api/strategy/*,
  /api/attack/*/toggle, /api/coc/proxy) implement CSRF protection. Flask-WTF or a
  custom CSRF token are absent. The authentication mechanism relies solely on the
  session cookie, which the browser automatically includes on cross-origin requests.
  Combined with the wildcard CORS setting (see MED-002), this creates a classic
  CSRF vulnerability.

Evidence:
  No import of flask_wtf or CSRFProtect anywhere in web/app.py.
  No X-CSRFToken header check in any endpoint.
  socketio = SocketIO(app, cors_allowed_origins="*")  -- covered in MED-002

  A malicious page can issue:
    fetch("http://<bot-host>:5000/api/start",
          {method:"POST", body:JSON.stringify({mode:"attack"}),
           credentials:"include", headers:{"Content-Type":"application/json"}})

Impact:
  Any webpage visited by the authenticated operator can start/stop the bot, save
  arbitrary config YAML, trigger attacks, replay strategies, or proxy CoC API
  requests -- all without the operator's consent.

Remediation:
  Option A (recommended): Add Flask-WTF CSRF protection:
    from flask_wtf.csrf import CSRFProtect
    csrf = CSRFProtect(app)

  Option B: Check a custom header that browsers do not send cross-origin without
  CORS permission:
    def check_csrf():
        if request.method in ("POST", "PUT", "DELETE", "PATCH"):
            if request.headers.get("X-Requested-With") != "XMLHttpRequest":
                abort(403)

  Option C: Use SameSite=Strict on the session cookie (partial mitigation only).

References:
  CWE-352: Cross-Site Request Forgery
  OWASP A01:2021 - Broken Access Control
  OWASP CSRF Prevention Cheat Sheet

[MED-002] WILDCARD CORS POLICY ON FLASK-SOCKETIO
Severity:    MEDIUM
File(s):     web/app.py (line 42)
Category:    CORS Misconfiguration

Description:
  SocketIO is initialized with cors_allowed_origins="*", which permits any origin
  to establish a WebSocket connection. This means any webpage -- regardless of its
  domain -- can connect to the SocketIO namespace, receive live screenshots, log
  messages, and bot stats without any origin restriction. The on_connect handler
  does check session.get("logged_in"), which prevents unauthenticated data access,
  but the wildcard origin allows cross-origin XHR/fetch requests to the REST API
  endpoints as well (browsers check CORS for cross-origin fetch).

Evidence:
  web/app.py line 42:
    socketio = SocketIO(app, cors_allowed_origins="*")

Impact:
  Cross-origin JavaScript can make authenticated requests to all REST endpoints
  when combined with the missing CSRF tokens (MED-001). The wildcard also disables
  cookie-based CORS protection, since credentials: "include" plus a wildcard
  CORS origin is actually rejected by modern browsers -- but any misconfiguration
  or older browser may behave differently. The combination is a systemic risk.

Remediation:
  Restrict to the exact origin(s) the dashboard is served from:
    socketio = SocketIO(app, cors_allowed_origins=["http://localhost:5000",
                                                    "http://127.0.0.1:5000"])
  Or derive dynamically from a HOST config value.

References:
  CWE-942: Permissive Cross-domain Policy with Untrusted Domains
  OWASP A05:2021 - Security Misconfiguration

[MED-003] PATH TRAVERSAL IN STRATEGY FILE NAME -- ARBITRARY JSON WRITE
Severity:    MEDIUM
File(s):     web/app.py (lines 302-309), bot/actions/strategy_recorder.py (lines 45-67)
Category:    Path Traversal / Arbitrary File Write

Description:
  The strategy name provided by the client via /api/strategy/record/stop is used
  directly to construct a file path with no sanitisation. An attacker who is
  authenticated (or exploits the CSRF vulnerability above) can supply a name
  containing directory traversal sequences to write a JSON file at any location
  writable by the bot process.

Evidence:
  web/app.py line 305:
    name = request.json.get("name", "default") if request.is_json else "default"
    filepath = bot.strategy_recorder.stop_recording(name)

  strategy_recorder.py line 53:
    filepath = os.path.join(STRATEGIES_DIR, f"{name}.json")
    ...
    with open(filepath, "w") as f:
        json.dump(strategy, f, indent=2)

  Payload: {"name": "../../web/templates/index"}
  Result:  strategies/../../web/templates/index.json  ==  web/templates/index.json
           (overwrites the index template with attacker-controlled JSON)

  A more targeted payload: {"name": "../config"}
  Result:  Overwrites config.yaml with JSON (disrupts YAML parsing on next load)

Impact:
  Authenticated write of attacker-controlled JSON to any path accessible to the
  process. This can corrupt templates, overwrite config files, or -- on writable
  systems with Python/module paths in scope -- potentially lead to code execution
  via a malicious .py file in a module directory.

Remediation:
  Sanitize the strategy name before use:
    import re
    def _safe_strategy_name(name: str) -> str:
        # Allow only alphanumeric, hyphen, underscore -- max 64 chars
        clean = re.sub(r'[^a-zA-Z0-9_\-]', '_', name)[:64]
        if not clean:
            clean = "default"
        return clean

  Additionally verify the resolved path stays within STRATEGIES_DIR:
    base = Path(STRATEGIES_DIR).resolve()
    filepath = (base / f"{name}.json").resolve()
    if not str(filepath).startswith(str(base) + os.sep):
        raise ValueError("Invalid strategy name")

References:
  CWE-22: Improper Limitation of a Pathname to a Restricted Directory
  OWASP A03:2021 - Injection

[MED-004] STRATEGY REPLAY ACCEPTS ARBITRARY UNVALIDATED FILE NAME
Severity:    MEDIUM
File(s):     web/app.py (lines 352-359), bot/actions/strategy_recorder.py (lines 73-109)
Category:    Path Traversal / Replay Attack

Description:
  The /api/strategy/replay endpoint accepts a strategy name from the client and
  constructs the file path with the same unsanitised join as stop_recording(). An
  attacker can supply a traversal sequence to read and replay any JSON file on the

  replayed at any time -- not just during an active attack -- strategies represent
  a persistent replay attack vector: credentials recorded inadvertently during
  "strategy recording" mode (e.g. a PIN entry visible on screen) can be replayed

    name = request.json.get("name", "default") if request.is_json else "default"

    filepath = os.path.join(STRATEGIES_DIR, f"{name}.json")

    with open(filepath, "r") as f:

[MED-005] COC API PROXY DOES NOT RESTRICT THE TARGET HOST -- SSRF RISK
Severity:    MEDIUM
File(s):     web/app.py (lines 398-443)
Category:    Server-Side Request Forgery (SSRF)

Description:
  The /api/coc/proxy endpoint constructs the target URL by prepending the fixed
  string "https://api.clashofclans.com/v1" to a user-supplied path parameter. The
  only validation is that path must start with "/". This prevents redirecting to a
  completely different host, but does not prevent certain SSRF sub-cases:
  1. The path parameter can include URL-encoded sequences or unusual characters
     that urllib might resolve unexpectedly.
  2. The token is transmitted by the client and relayed by the server, meaning
     the server acts as a credentialed proxy for any valid CoC API path. If a
     future API version includes a redirect to a different host, the proxy would
     follow it.
  3. More importantly: the API token is sent from the browser to this server-side
     proxy and then forwarded to Supercell. The token is visible in the request
     body in browser devtools and in server logs if request bodies are logged.

Evidence:
  web/app.py line 419:
    url = "https://api.clashofclans.com/v1" + path

  Minimal path check:
    if not path.startswith("/"):
        return jsonify({"error": "Invalid path"}), 400

  No allowlist of valid path prefixes, no check for null bytes or encoded
  traversal, no URL validation beyond the leading slash.

Impact:
  Limited SSRF risk due to hardcoded host prefix. Primary risk is that a
  malicious path could exploit edge cases in urllib.request.urlopen URL parsing.
  The secondary risk is CoC API token exposure: the token is sent client-side in
  localStorage (MED-006) and could be logged by the server.

Remediation:
  Enforce an allowlist of valid path prefixes:
    ALLOWED_PREFIXES = ("/clans/", "/players/", "/leagues/", "/locations/",
                        "/labels/", "/goldpass/", "/warleagues/",
                        "/builderbaseleagues/", "/clanwarleagues/")
    if not any(path.startswith(p) for p in ALLOWED_PREFIXES):
        return jsonify({"error": "Path not allowed"}), 400

  Also validate path contains no null bytes: if '\x00' in path: abort(400)

References:
  CWE-918: Server-Side Request Forgery
  OWASP A10:2021 - Server-Side Request Forgery

[MED-006] COC API TOKEN STORED IN BROWSER LOCALSTORAGE -- XSS EXTRACTION RISK
Severity:    MEDIUM
File(s):     web/templates/index.html (lines 753-766)
Category:    Sensitive Data Exposure / XSS

Description:
  The CoC API key entered by the user is persisted to localStorage:
    localStorage.setItem('coc_api_key', k)
  localStorage is accessible to any JavaScript running on the same origin. If an
  XSS vulnerability is ever introduced (see MED-007), an attacker's injected script
  can silently exfiltrate the API token. Additionally, the token is transmitted to
  the server as a plain string in the POST body of every /api/coc/proxy request,
  making it visible in browser network devtools and susceptible to interception on
  unencrypted HTTP connections.

Evidence:
  index.html line 756:
    localStorage.setItem('coc_api_key', k);
  index.html line 765:
    const savedKey = localStorage.getItem('coc_api_key');

Impact:
  A stolen CoC API token allows an attacker to query the Clash of Clans API on
  behalf of the user (read clan/player data, verify tokens). The API key is tied
  to specific IP addresses by Supercell's developer portal, so exploitation from
  an arbitrary IP is blocked -- but from the same IP it is fully functional.

Remediation:
  Store the API key server-side (encrypted in the Flask session or an env variable)
  rather than in localStorage. The browser should only send a request identifier,
  not the raw token. If client-side storage is required, use sessionStorage (which
  clears on tab close) rather than localStorage.

References:
  CWE-922: Insecure Storage of Sensitive Information
  OWASP A02:2021 - Cryptographic Failures

[MED-007] POTENTIAL XSS IN DONATION HISTORY RENDERING VIA UNSANITIZED TROOP NAME
Severity:    MEDIUM
File(s):     web/templates/index.html (lines 388-393), bot/actions/donator.py
Category:    Cross-Site Scripting (XSS)

Description:
  The donation history table is rendered by inserting troop names directly into
  innerHTML without HTML-escaping:
    tbody.innerHTML = recent.map(h => {
        return `<tr><td>${timeStr}</td><td>${h.troop}</td></tr>`;
    }).join('');
  The troop name (h.troop) comes from the server via the SocketIO stats event and
  originates from donator.py. If a template file name or config entry for troop
  names contains HTML/JS characters -- or if the stats data is ever sourced from
  an untrusted input -- this results in stored XSS within the dashboard. The risk
  is low today because troop names come from config.yaml, but the pattern is
  architecturally unsafe.

Evidence:
  index.html line 391-392:
    return `<tr><td>${timeStr}</td><td>${h.troop}</td></tr>`;
  (no escaping of h.troop before insertion into innerHTML)

Impact:
  If h.troop is ever set to a value like <img src=x onerror=alert(1)>, it executes
  in the authenticated operator's browser. With the API token in localStorage
  (MED-006), this becomes a complete token theft chain.

Remediation:
  Use textContent instead of innerHTML for the troop cell, or escape with a helper:
    function escH(s){return String(s).replace(/&/g,'&amp;').replace(/</g,'&lt;').replace(/>/g,'&gt;');}

[HIGH-001] CONFIG SAVE ENDPOINT ALLOWS ARBITRARY YAML INJECTION -- BEHAVIOR OVERRIDE
Severity:    HIGH
File(s):     web/app.py (lines 237-260)
Category:    Injection / Configuration Tampering

Description:
  The /api/config POST endpoint accepts an arbitrary YAML string from the
  authenticated client and writes it directly to the config file after only a
  yaml.safe_load() validity check. While safe_load prevents YAML deserialization
  code execution (unlike yaml.load with full_load), a maliciously crafted but
  syntactically valid config can change any bot behaviour:
  - Set safety.dry_run to false (enable live taps)
  - Set attack.min_loot.gold/elixir to 0 (attack every base found)
  - Set safety.max_attacks to 999999 (run indefinitely)
  - Change device.serial to a different attached device
  - Set logging.file to an arbitrary path to write log output anywhere the
    process can write

  Combined with MED-001 (CSRF), an unauthenticated cross-site attacker can trigger
  this endpoint without the operator's knowledge.

Evidence:
  web/app.py lines 244-249:
    new_config_text = request.json.get("config", "")
    yaml.safe_load(new_config_text)   # only validates YAML syntax
    with open(config_path, "w") as f:
        f.write(new_config_text)      # unrestricted overwrite

Impact:
  Full control over bot behaviour and indirect control over the ADB-connected
  Android device. Log output redirection to system paths (e.g. /etc/cron.d/ on
  Linux) could be escalated to code execution if the process has write permission
  there.

Remediation:
  1. Implement a schema validator that checks that only expected keys are present
     and that values are within acceptable types and ranges before writing:
       SCHEMA = {

           "attack": {"min_loot": {"gold": int, "elixir": int}, ...},

       }
  2. Restrict the logging.file value to within the project logs/ directory.
  3. Add CSRF protection (MED-001) to prevent cross-site config tampering.

  CWE-15: External Control of System or Configuration Setting

[HIGH-002] WEBSOCKET LOG STREAM LEAKS FULL APPLICATION LOGS TO ALL CONNECTED CLIENTS
Severity:    HIGH
File(s):     web/app.py (lines 122-133, 364-373)
Category:    Information Disclosure / Authorization

Description:
  The WebLogHandler captures all log records from the root logger and pushes them
  to every connected SocketIO client via socketio.emit("log", ...). There is no
  per-connection authorization check on the background emitter task -- the emit is
  global. The on_connect handler does verify session.get("logged_in"), which
  disconnects unauthenticated WebSocket connections, but the background emitter
  uses socketio.emit() (broadcast to all rooms/namespaces) rather than emit to
  a specific authenticated session ID.

  More critically: the log stream is sent to the root logger handler. If debug
  logging is enabled (--debug flag), this stream includes:
  - Tap coordinates for every tap sent to the device
  - OCR-read loot values
  - Template match confidences and paths
  - Device resolution
  - ADB stderr output (which may include device identifiers or error details)

Evidence:
  web/app.py lines 130-132:
    socketio.emit("log", {"message": msg})   # broadcast to ALL clients

  adb_controller.py line 117:
    logger.debug("Tap at (%d, %d)", x, y)   # sent to web if debug enabled

  vision.py line 103:
    logger.debug("Match '%s': (%.3f) at (%d, %d)...", template_path, ...)

Impact:
  If an attacker maintains a WebSocket connection (which requires passing the
  session check), they receive a real-time stream of all bot activity including
  precise tap coordinates and device state. If the on_connect session check is
  bypassed (e.g. via the CSRF/CORS issues above), this becomes a broadcast to
  unauthenticated clients.

  1. Emit logs only to the specific session that is authenticated, not globally:
       socketio.emit("log", {"message": msg}, to=request.sid)

  3. Consider a dedicated log room that clients join after authentication rather

[HIGH-003] SINGLE-USER AUTHENTICATION WITH NO SESSION EXPIRY OR MFA
Severity:    HIGH
File(s):     web/app.py (lines 47-113), users.json
Category:    Authentication / Authorization

Description:
  The authentication model is a single hardcoded user account stored in users.json
  with no support for:
  - Multi-factor authentication
  - Account lockout after failed attempts (see INFO-004)
  - Password reset / rotation mechanism
  - Session expiry (sessions persist until SECRET_KEY changes or session.clear() is
    called; there is no idle timeout or absolute session lifetime configured)
  - Multiple users / roles

  Once an attacker obtains the session cookie (via XSS, network sniffing, or CSRF),
  there is no second factor to prevent full access. Flask's default session has no
  expiry unless SESSION_PERMANENT and PERMANENT_SESSION_LIFETIME are configured.

Evidence:
  web/app.py lines 64-70: login_required checks only session.get("logged_in") --
  a single boolean with no timestamp, IP binding, or rotation.

  No PERMANENT_SESSION_LIFETIME is configured; sessions are permanent unless
  explicitly invalidated.

  users.json: single user account with no lockout state.

Impact:
  A stolen session cookie provides unrestricted access forever (until the server
  restarts with a new secret or the user manually logs out). Any of the HIGH/CRIT
  findings above become easier to exploit once a session is stolen.

Remediation:
  1. Set session lifetime:
       from datetime import timedelta
       app.config["PERMANENT_SESSION_LIFETIME"] = timedelta(hours=8)
       app.config["SESSION_PERMANENT"] = True
  2. Rotate session ID on login (Flask does not do this automatically):
       session.clear()
       session["logged_in"] = True
       session["login_time"] = time.time()
  3. Store last-activity timestamp in session and enforce idle timeout.
  4. Document that the password must be strong (>= 16 chars, random).

References:
  CWE-613: Insufficient Session Expiration
  OWASP A07:2021 - Identification and Authentication Failures

[HIGH-004] MONKEY COMMAND USED FOR APP RELAUNCH -- ARGUMENT INJECTION POSSIBLE
Severity:    HIGH
File(s):     bot/core.py (lines 323-324)
Category:    Command Injection

Description:
  The _relog() method launches the Clash of Clans app using the `adb shell monkey`
  command with the package name taken from self._coc_package, which is hardcoded
  as a string literal in __init__. At present this is safe because the value is
  constant. However, the monkey command is a high-risk pattern:

  1. The monkey command is a generic Android event injector. With the right flags
     it can simulate arbitrary UI input across any application on the device --
     far beyond what is needed to simply launch an app. If self._coc_package were
     ever sourced from config (e.g., config.device.package), a malicious config
     could inject additional shell arguments.

  2. The specific command is:
       adb shell monkey -p <package> -c android.intent.category.LAUNCHER 1
     If package were ever attacker-controlled (e.g., from config.yaml after HIGH-001
     exploitation), an injection like:
       com.supercell.clashofclans -c android.intent.category.LAUNCHER 1; am start ...
     would execute arbitrary Android shell commands because adb shell passes the
     concatenated argument string to /bin/sh on the device.

  Note: The arguments are passed as a list to subprocess.run(), so the HOST shell
  is not invoked (no shell=True). However, because this is "adb shell monkey ...",
  the arguments after "shell" are assembled into a shell command string on the
  Android device side, so injection into the package name parameter IS effective
  on the remote device shell.

Evidence:
  bot/core.py lines 323-324:
    self.adb._run("shell", "monkey", "-p", self._coc_package, "-c",
                   "android.intent.category.LAUNCHER", "1")

  self._coc_package = "com.supercell.clashofclans"  (line 57, hardcoded)

  If an attacker gains control of _coc_package (via HIGH-001 + future config
  sourcing of the package name), the payload:
    "com.supercell.clashofclans; am broadcast -a com.evil.action"
  would execute on the Android device.

Impact:
  If self._coc_package is ever sourced from config or user input, this is a
  remote command injection into the Android shell. Even as-is, using monkey for
  app launch is unnecessarily dangerous -- a safer alternative exists.

Remediation:
  Replace monkey with the explicit am start command, which only launches a single
  activity and does not have monkey's broad event-injection capabilities:
    self.adb._run("shell", "am", "start",
                  "-a", "android.intent.action.MAIN",
                  "-c", "android.intent.category.LAUNCHER",
                  "-n", f"{self._coc_package}/.MainActivity")

  Never source the package name from config without strict validation against an
  allowlist of known CoC package names.

References:
  CWE-78: OS Command Injection
  Android adb shell monkey documentation

[CRIT-001] AUTHENTICATED CONFIG SAVE + STRATEGY PATH TRAVERSAL = RCE CHAIN
Severity:    CRITICAL
File(s):     web/app.py (lines 237-260, 302-309, 352-359),
             bot/actions/strategy_recorder.py (lines 45-67, 73-109),
             bot/config_loader.py (lines 95-106)
Category:    Remote Code Execution (Chained)

Description:
  A multi-step exploit chain exists that allows an authenticated attacker (or an
  unauthenticated attacker exploiting the CSRF vulnerability in MED-001) to achieve
  arbitrary file writes and potentially arbitrary code execution on the machine
  running the bot. The chain works as follows:

  EXPLOIT PATH A -- Direct path traversal + file write (requires only MED-003):
    1. Start recording: POST /api/strategy/record/start
    2. Add at least one tap: POST /api/strategy/tap  {"x":1,"y":1}
    3. Save with traversal name:
         POST /api/strategy/record/stop
         {"name": "../../web/templates/index"}
       Result: web/templates/index.json is overwritten with attacker-controlled
       JSON containing the recorded tap events. On the next page load, Flask's
       Jinja2 template loader may attempt to parse the corrupted file and throw
       errors, causing a denial-of-service.

  EXPLOIT PATH B -- Config write redirects log output + strategy replay reads it:
    1. POST /api/config with YAML setting logging.file to ../strategies/payload.json
    2. Trigger bot activity to write log lines into strategies/payload.json
    3. Craft log output to contain a valid JSON structure with "events" array via
       a specially timed log message
    4. POST /api/strategy/replay {"name": "payload"}
       The strategy_recorder reads strategies/payload.json and executes any
       "tap_down" typed events as ADB taps on the device.
       (Note: replay at line 105 checks event["type"] == "tap_down", not "tap",
       so the standard recorded format won't execute -- but a crafted file with
       "tap_down" events would.)

  EXPLOIT PATH C -- Config write + log file path to system directory (Linux):
    1. POST /api/config with logging.file set to /etc/cron.d/coc_bot
    2. Start the bot; log lines are written to /etc/cron.d/coc_bot
    3. Cron parses the file; comment-like log lines are ignored, but a crafted
       log message that matches cron syntax would execute at scheduled intervals.
       This requires the bot process to have write access to /etc/cron.d/, which
       is unlikely but possible on misconfigured systems.

  The critical architectural observation is that HIGH-001 (unrestricted config
  write), MED-003 (unsanitised strategy name), and MED-004 (unsanitised replay
  name) can be chained in multiple ways to achieve write primitives anywhere the
  process has filesystem access.

Evidence:
  strategy_recorder.py line 53:
    filepath = os.path.join(STRATEGIES_DIR, f"{name}.json")
    # STRATEGIES_DIR = "strategies" (relative to cwd, not absolute)

  web/app.py lines 244-249:
    new_config_text = request.json.get("config", "")
    yaml.safe_load(new_config_text)   # only syntax check, no schema validation
    with open(config_path, "w") as f:
        f.write(new_config_text)

  web/app.py lines 250-254:
    if not bot.running:
        config = load_config(config_path)
        bot = Bot(config)   # Bot reinitialised with attacker config immediately

  Proof of concept for Exploit Path A (curl commands):
    curl -b "session=<valid_cookie>" -X POST http://localhost:5000/api/strategy/record/start
    curl -b "session=<valid_cookie>" -X POST http://localhost:5000/api/strategy/tap \
         -H "Content-Type: application/json" -d '{"x":1,"y":1}'
    curl -b "session=<valid_cookie>" -X POST http://localhost:5000/api/strategy/record/stop \
         -H "Content-Type: application/json" -d '{"name":"../../web/templates/index"}'

Impact:
  - Corrupt web templates -> denial of service for the dashboard
  - Overwrite config.yaml with attacker-controlled values -> full bot behaviour control
  - On Linux: redirect log output to system directories -> potential persistent RCE
  - Execute arbitrary ADB taps on the connected Android device at attacker-specified
    coordinates (device account takeover, in-game actions)

Remediation (in priority order):
  1. IMMEDIATE: Sanitize strategy names (alphanumeric + hyphen/underscore only,
     max 64 chars). Validate resolved path is within STRATEGIES_DIR:
       import re
       from pathlib import Path

       def _safe_strategy_name(name: str) -> str:
           clean = re.sub(r'[^a-zA-Z0-9_\-]', '_', str(name))[:64]
           return clean or "default"

       def _safe_strategy_path(name: str) -> Path:
           base = Path(STRATEGIES_DIR).resolve()
           candidate = (base / f"{_safe_strategy_name(name)}.json").resolve()
           if not str(candidate).startswith(str(base) + os.sep):
               raise ValueError("Path traversal attempt detected")
           return candidate

  2. IMMEDIATE: Add CSRF protection to all POST endpoints (see MED-001).

  3. SHORT-TERM: Add YAML schema validation to /api/config that restricts:
     - logging.file must be within the logs/ subdirectory
     - device.serial must match the ADB serial validation regex
     - All numeric fields must be within reasonable bounds

  4. SHORT-TERM: Run the bot process as a dedicated low-privilege OS user with
     write permissions scoped to: logs/, strategies/, config.yaml only.

  5. LONG-TERM: Consider making config changes require a bot restart confirmation
     rather than immediate hot-reload.

References:
  CWE-22: Improper Limitation of a Pathname to a Restricted Directory
  CWE-78: OS Command Injection
  CWE-94: Code Injection
  CWE-20: Improper Input Validation
  OWASP A03:2021 - Injection
  OWASP A01:2021 - Broken Access Control
